# JobPulse — Full Project Specification
> **For AI agents**: This document is the single source of truth for the JobPulse platform. Read it fully before writing any code. Every decision, schema, naming convention, architecture pattern, and build order is defined here. Do not deviate from what is specified.

---

## 1. What This Project Is

JobPulse is a **Job Market Intelligence Platform** built with **ASP.NET MVC Core**. It is NOT a job application platform. Users do not apply to jobs through it. The purpose is to help job seekers and students understand market demand — which roles are in demand, what skills are required, and what salaries look like — so they can make informed career decisions.

**The data is provided externally** by a data engineer teammate who scrapes job boards and populates a shared SQL Server database. The web team (this project) only reads from job data tables and writes only to user-owned tables (auth, profiles, salary reports, saved jobs).

**Tech stack (non-negotiable):**
- Backend: ASP.NET MVC Core (.NET 8)
- ORM: Entity Framework Core
- Auth: ASP.NET Core Identity
- Database: SQL Server
- Frontend: Razor Views (.cshtml) + Tailwind CSS (CDN) + Flowbite components + HTMX + Chart.js

## 2. Database Schema

### Rules
- All primary keys on user-owned tables are `int` with identity, except `ApplicationUser` which inherits `IdentityUser` (string GUID).
- Job data tables (`Jobs`, `Skills`, `JobSkills`, `DemandSnapshots`) are owned by the data pipeline. The web app only reads from them. Never run `UPDATE` or `DELETE` on these tables.
- Skills must be stored as a canonical normalized list. The pipeline maps raw scraped skill strings to canonical names. There is one row per skill in the `Skills` table.

### Canonical Skill List (agreed with data engineer)
```
JavaScript, TypeScript, Python, Java, C#, .NET, React, Angular, Vue,
Node.js, SQL, PostgreSQL, MongoDB, Docker, AWS, Azure, Git,
HTML/CSS, PHP, Kotlin, Swift, Go, Rust, Flutter, GraphQL
```

## 5. ApplicationDbContext

```csharp
using Microsoft.AspNetCore.Identity.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore;

public class ApplicationDbContext : IdentityDbContext<ApplicationUser>
{
    public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options)
        : base(options) { }

    public DbSet<Job> Jobs => Set<Job>();
    public DbSet<Skill> Skills => Set<Skill>();
    public DbSet<JobSkill> JobSkills => Set<JobSkill>();
    public DbSet<UserSkill> UserSkills => Set<UserSkill>();
    public DbSet<DemandSnapshot> DemandSnapshots => Set<DemandSnapshot>();
    public DbSet<SalaryReport> SalaryReports => Set<SalaryReport>();
    public DbSet<SavedJob> SavedJobs => Set<SavedJob>();

    protected override void OnModelCreating(ModelBuilder builder)
    {
        base.OnModelCreating(builder);

        // Jobs — read-only from pipeline, map to existing table name
        builder.Entity<Job>().ToTable("Jobs");
        builder.Entity<Skill>().ToTable("Skills");
        builder.Entity<JobSkill>().ToTable("JobSkills");
        builder.Entity<DemandSnapshot>().ToTable("DemandSnapshots");

        // Unique constraint: one user can't save same job twice
        builder.Entity<SavedJob>()
            .HasIndex(s => new { s.UserId, s.JobId })
            .IsUnique();

        // Unique constraint: one user can't have same skill twice
        builder.Entity<UserSkill>()
            .HasIndex(u => new { u.UserId, u.SkillId })
            .IsUnique();

        // Decimal precision
        builder.Entity<Job>()
            .Property(j => j.SalaryMin).HasPrecision(10, 2);
        builder.Entity<Job>()
            .Property(j => j.SalaryMax).HasPrecision(10, 2);
        builder.Entity<SalaryReport>()
            .Property(s => s.Salary).HasPrecision(10, 2);
    }
}
```

using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));

app.UseStaticFiles();
app.UseRouting();
app.UseAuthentication();

    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

## 8. Controllers

### Architecture Rule
Controllers must be thin. The pattern is always:
1. Receive HTTP request
2. Call a service method
3. Map result to a ViewModel
4. Return View(viewModel)

Never write business logic (filtering, ranking, matching) inside a controller method.

### HomeController.cs
```csharp
[HttpGet]
public async Task<IActionResult> Index()
{
    var topRoles = await _rankingService.GetTopRolesAsync(10);
    var topSkills = await _rankingService.GetTopSkillsAsync(10);
    var recentJobs = await _jobService.GetRecentAsync(6);

    List<Job>? personalizedJobs = null;
    if (User.Identity?.IsAuthenticated == true)
    {
        var userId = _userManager.GetUserId(User)!;
        personalizedJobs = await _recommendationService.GetRecommendedJobsAsync(userId, 6);
    }

    var vm = new HomeViewModel
    {
        TopRoles = topRoles,
        TopSkills = topSkills,
        RecentJobs = recentJobs,
        PersonalizedJobs = personalizedJobs
    };
    return View(vm);
}
```

### JobsController.cs — key actions
```csharp
// GET /Jobs?query=backend&location=Cairo&page=1
[HttpGet]
public async Task<IActionResult> Index(string? query, string? location,
    string? experienceLevel, decimal? salaryMin, decimal? salaryMax, int page = 1)
{
    var (jobs, totalCount) = await _jobService.SearchAsync(
        query, location, experienceLevel, salaryMin, salaryMax, null, page);

    var vm = new JobSearchViewModel
    {
        Jobs = jobs,
        TotalCount = totalCount,
        CurrentPage = page,
        Query = query,
        Location = location
    };

    // Return partial if HTMX request (live search)
    if (Request.Headers.ContainsKey("HX-Request"))
        return PartialView("_JobResults", vm);

    return View(vm);
}

// GET /Jobs/Detail/123
[HttpGet]
public async Task<IActionResult> Detail(long id)
{
    var job = await _jobService.GetByIdAsync(id);
    if (job == null) return NotFound();

    bool isSaved = false;
    if (User.Identity?.IsAuthenticated == true)
    {
        var userId = _userManager.GetUserId(User)!;
        isSaved = await _jobService.IsJobSavedAsync(userId, id);
    }

    var vm = new JobDetailViewModel { Job = job, IsSaved = isSaved };
    return View(vm);
}

// POST /Jobs/Save — requires auth
[HttpPost]
[Authorize]
public async Task<IActionResult> Save(long jobId)
{
    var userId = _userManager.GetUserId(User)!;
    await _jobService.SaveJobAsync(userId, jobId);
    return Ok();
}
```

### AccountController.cs — key actions
```csharp
[HttpPost]
public async Task<IActionResult> Register(RegisterViewModel vm)
{
    if (!ModelState.IsValid) return View(vm);

    var user = new ApplicationUser
    {
        UserName = vm.Email,
        Email = vm.Email,
        DisplayName = vm.DisplayName,
        CreatedAt = DateTime.UtcNow
    };

    var result = await _userManager.CreateAsync(user, vm.Password);
    if (result.Succeeded)
    {
        await _signInManager.SignInAsync(user, isPersistent: false);
        return RedirectToAction("Index", "Profile");
    }

    foreach (var error in result.Errors)
        ModelState.AddModelError(string.Empty, error.Description);

    return View(vm);
}
```

### HTMX Live Search Pattern
In `Views/Jobs/Index.cshtml`, the search input triggers a partial view reload:
```html
<input type="text"
       name="query"
       hx-get="/Jobs"
       hx-trigger="keyup changed delay:400ms"
       hx-target="#job-results"
       hx-include="[name='location'],[name='experienceLevel'],[name='salaryMin'],[name='salaryMax']"
       placeholder="Search jobs..."
       class="..." />

<div id="job-results">
    @await Html.PartialAsync("_JobResults", Model)
</div>
```

In `JobsController.Index`, detect HTMX and return partial:
```csharp
if (Request.Headers.ContainsKey("HX-Request"))
    return PartialView("_JobResults", vm);
return View(vm);
```

The project will be built in 5 phases. Do not start a phase until the previous one has a working checkpoint.

### Phase 1 — Foundation
**Goal:** Database exists, EF Core works, Identity configured.

Steps:
1. Create ASP.NET MVC Core project targeting .NET 8
2. Install NuGet packages: `Microsoft.EntityFrameworkCore.SqlServer`, `Microsoft.EntityFrameworkCore.Tools`, `Microsoft.AspNetCore.Identity.EntityFrameworkCore`
3. Write all entity classes (Section 4)
4. Write `ApplicationDbContext` (Section 5)
5. Configure `Program.cs` (Section 6)
6. Set connection string in `appsettings.json`
7. Run: `dotnet ef migrations add InitialCreate` then `dotnet ef database update`
8. Seed 100 fake jobs covering 5 cities, 15 roles, 15 skills

**Checkpoint:** A test controller action can query `db.Jobs.Take(5).ToList()` and return JSON without errors.

---

### Phase 2 — Core Read-Only Features
**Goal:** Anonymous user can search jobs and view rankings.

Steps:
1. Implement `JobService` with `SearchAsync` using EF Core LINQ with optional filters
2. Implement `RankingService` with `GetTopRolesAsync` (GROUP BY Title ORDER BY COUNT DESC) and `GetTopSkillsAsync`
3. Build `JobsController` (Index with search/filter, Detail)
4. Build `RankingsController` (Index, RoleDetail)
5. Build `_Layout.cshtml` with all CDN imports
6. Build Views: Home/Index, Jobs/Index, Jobs/Detail, Rankings/Index, Rankings/RoleDetail
7. Add Chart.js bar chart to Rankings/Index
8. Add HTMX to search input on Jobs/Index

**Checkpoint:** Anonymous user can search for "Python developer in Cairo", see filtered results, click a job, see the detail page. Rankings page shows top 10 roles as a horizontal bar chart.

---

### Phase 3 — Auth + User Accounts
**Goal:** Users can register, log in, and save jobs.

Steps:
1. Build `AccountController` (Register GET/POST, Login GET/POST, Logout POST)
2. Build `Views/Account/Register.cshtml` and `Login.cshtml` with Flowbite form components
3. Build `ProfileController` (Index GET to view profile, Edit POST to update skills)
4. Build `Views/Profile/Index.cshtml` with multi-select skill picker
5. Update `_Navbar.cshtml` to show user display name and logout when authenticated
6. Add Save button to job cards and detail page (posts to `/Jobs/Save`, guarded by `[Authorize]`)
7. Build saved jobs list at `Profile/SavedJobs`

**Checkpoint:** New user registers, is redirected to profile setup, selects 5 skills, saves 3 jobs, sees them in saved jobs list.

---

### Phase 4 — Personalization + Salary Contribution
**Goal:** Logged-in users see personalized feed. Any user can submit salary data.

Steps:
1. Implement `RecommendationService.GetRecommendedJobsAsync`: get user's skill IDs → query jobs where JobSkills overlaps → order by overlap count descending → take top N
2. Add personalized "Jobs for you" section to `Home/Index.cshtml` (visible only when authenticated)
3. Build `SalaryController` (Submit GET/POST)
4. Build `Views/Salary/Submit.cshtml` with form validation
5. Blend `SalaryReports` data into `RankingService.GetSalaryRangeAsync` (average with pipeline data)

**Checkpoint:** Logged-in user with React + Python skills sees React and Python jobs first on home page. Salary form submits successfully and data appears in DB.

---

### Phase 5 — Polish (if time permits)
**Goal:** Platform feels like a real product.

Steps:
1. Add demand trend line chart to `Rankings/RoleDetail` using `DemandSnapshots` data
2. Add trending skills section to home page
3. Ensure all pages are mobile-responsive using Tailwind responsive prefixes (`sm:`, `md:`, `lg:`)
4. Add loading spinners on HTMX requests using `htmx:beforeRequest` and `htmx:afterRequest` events
5. Add empty state messages when search returns 0 results
6. Add pagination controls to search results

**Checkpoint:** Platform looks professional. Mobile layout works. Empty states are handled gracefully.

## 12. Data Rules (Agreed with Data Engineer)

| Rule | Detail |
|---|---|
| Skills are normalized | Pipeline maps all skill strings to canonical names from the agreed list |
| Web app never writes to pipeline tables | `Jobs`, `Skills`, `JobSkills`, `DemandSnapshots` are read-only from the web app |
| `IsActive = false` jobs | Still in DB for history but filtered out of search results by default |
| Salary data blending | `RankingService` queries both `Jobs.SalaryMin/Max` (pipeline) and `SalaryReports.Salary` (user-submitted), averages them together |
| Snapshot frequency | `DemandSnapshots` has one row per role per location per week |

## 13. Naming Conventions

| Thing | Convention | Example |
|---|---|---|
| Controllers | PascalCase + Controller suffix | `JobsController` |
| Actions | PascalCase verb | `Index`, `Detail`, `Submit` |
| Views | Match action name | `Index.cshtml`, `Detail.cshtml` |
| Partial views | Underscore prefix | `_JobCard.cshtml` |
| Services | Interface + Implementation | `IJobService`, `JobService` |
| ViewModels | PascalCase + ViewModel suffix | `JobSearchViewModel` |
| Entity classes | Singular PascalCase | `Job`, `Skill`, `SavedJob` |
| DB table names | Plural PascalCase | `Jobs`, `Skills`, `SavedJobs` |
